# تقرير التحسينات والترقيات - نظام العمولات وZoonasd

تم تنفيذ مجموعة من التحسينات الاحترافية على لوحة المسوق ولوحة الإدارة مع التركيز على الجوانب الأمنية ودقة الحسابات.

## 1. آلية احتساب العمولات (التفصيل)

تم اعتماد نظام عمولة متدرج (Tiered Commission) يضمن العدالة ودقة الأرباح:

*   **العمولة بناءً على سعر المنتج:** يتم احتساب العمولة لكل منتج على حدة داخل الطلب، وليس على إجمالي الطلب. هذا يضمن تطبيق النسبة الصحيحة لكل قطعة.
*   **القواعد الحالية:**
    *   المنتجات التي سعرها **أقل من 100,000 ج.س**: العمولة هي **8%**.
    *   المنتجات التي سعرها **100,000 ج.س أو أكثر**: العمولة هي **6%**.
    *   **ملاحظة:** إذا حدد المدير عمولة ثابتة (Fixed Commission) لمنتج معين في لوحة المنتجات، فإنها تأخذ الأولوية على النسب المئوية.
*   **ثبات العمولة (Snapshotting):** عند إنشاء الطلب، يتم حساب العمولة فوراً وحفظ القيمة النهائية في جدول `affiliate_orders`.
    *   **لماذا؟** لحماية المسوق والمدير. إذا تغير سعر المنتج أو نسبة العمولة في المستقبل، تظل العمولات المسجلة للطلبات القديمة ثابتة كما كانت وقت الشراء.
*   **إجمالي الطلب:** في حالات نادرة (مثل فشل جلب بيانات منتج معين)، يتم تطبيق النسبة المتدرجة على إجمالي الطلب الصافي (بدون التوصيل) كإجراء احتياطي.

## 2. التحسينات الأمنية المنفذة

*   **تشفير كلمة مرور الإدارة (Hashing):**
    *   لا يتم تخزين كلمة المرور كنص عادي. تم استخدام خوارزمية **SHA-256**.
    *   يتم التحقق من كلمة المرور بالكامل في الخادم (Server-side) داخل وظائف Vercel API، مما يمنع محاولات التلاعب في الواجهة الأمامية.
*   **إخفاء مفاتيح Supabase:** تم إزالة جميع مفاتيح الربط (API Keys) من كود الواجهة الأمامية تماماً. التواصل مع قاعدة البيانات يتم حصراً عبر Proxy آمن (Vercel API) يستخدم متغيرات بيئة مشفرة.
*   **سياسات حماية البيانات (RLS):**
    *   تعديل سياسات `admin_settings` لمنع القراءة العامة لمفتاح كلمة المرور، مع السماح فقط بقراءة نسب العمولات.
    *   منع أي عمليات كتابة أو تعديل (INSERT, UPDATE, DELETE) من الواجهة العامة، وحصرها في المسؤول المصرح له بكلمة مرور.

## 3. التعديلات على الجداول والـ APIs

### الجداول المتأثرة (Database Schema):
*   `admin_settings`: جدول جديد (أو محدث) لتخزين الإعدادات (الحد الفاصل، النسب، كلمة المرور المشفرة).
*   `affiliate_orders`: تم التأكد من حفظ قيمة العمولة وقت الإنشاء.

### الـ API Endpoints:
*   `/api/orders`: يدعم الآن التحقق من كلمة المرور المشفرة للعمليات الإدارية.
*   `/api/admin-products`: يدعم الآن التحقق من كلمة المرور المشفرة.
*   `/api/admin-auth`: يقوم بالتحقق من كلمة المرور عبر مقارنة الـ Hash في الخادم.

## 4. ملخص خطوات الترقية (Migrations)

يجب تنفيذ الكود التالي في SQL Editor الخاص بـ Supabase لضمان عمل النظام:

```sql
-- 1. تحديث كلمة المرور إلى Hash (كلمة المرور الافتراضية هي zoona2025)
UPDATE admin_settings
SET value = '0e8f6b07e379770c480eb2817d5596a64334849ab8090fe870d79402312ca162'
WHERE key = 'admin_password';

-- 2. تأمين إعدادات المدير (RLS)
DROP POLICY IF EXISTS "Allow public read for commission settings" ON admin_settings;
CREATE POLICY "Public read non-sensitive settings" ON admin_settings
FOR SELECT
TO anon
USING (key IN ('commission_threshold', 'commission_low_rate', 'commission_high_rate'));

-- 3. تفعيل RLS على جميع الجداول الحساسة
ALTER TABLE admin_settings ENABLE ROW LEVEL SECURITY;
ALTER TABLE affiliate_users ENABLE ROW LEVEL SECURITY;
```
